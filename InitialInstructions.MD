### Задание 2: Scheduler (Interactive / Batch)
- Реализовали двухклассовый планировщик в ядре Phoenix-RTOS.
- `processClass` (0=Interactive, 1=Batch) хранится в `process_t`.
- `currentQuanta` хранится в `thread_t` (Interactive=1 тик, Batch=0x7FFFFFFF).
- Interactive: Round-Robin (переключение каждый тик).
- Batch: FIFO (работает до завершения, бесконечный квант).
- Interactive всегда вытесняет Batch (проверка `_hasReadyInteractive()` на каждом тике).
- Ключевой нюанс: проверяются только READY-потоки, спящие невидимы → Batch получает CPU когда все Interactive спят.
- Прерванный Batch возвращается в ГОЛОВУ очереди (FIFO-порядок сохраняется).
- Syscall: `setProcessClass(pid, class)` / `getProcessClass(pid)`.
- Тест: `schedtest <0|1>` — запускает фоновый процесс нужного класса. Проверка через `ps` + колонка TIME.

## Новое задание (текущее)

**Synchronization w/semaphores**

Реализовать FIFO-очередь на 16 элементов. Каждый элемент имеет тип (A/B/C) и значение (uint8_t). Очередь синхронизирует передачу между producer- и consumer-процессами:

- **3 producer-процесса** (по одному на каждый тип). Каждый создаёт 64-128 элементов своего типа. Без задержек.
- **2 consumer-процесса**:
  - Consumer 1: потребляет элементы типа A или B (по одному), но ТОЛЬКО если в очереди НЕТ элемента C.
  - Consumer 2: потребляет элементы всех трёх типов (сразу), если в очереди ЕСТЬ элемент C. Если в очереди ТОЛЬКО элементы C — потребляет C по одному.
- Решение на C/C++, используя **процессы, shared memory и семафоры в Linux** (не Phoenix).
- Подготовить демонстрацию.
- Для демонстраций преподавателю будет необходимо показать:
  - работоспособность программы со всеми имплементациями
  - работоспособность программы с delay для producers и без семафоров в имплементаций (можно либо сделать программу способный пережить простое закоментаривание имплементаций семафоров, либо подготовить уникальные флаги в аргументах запуска программы)
  - при запуске программы должно явно отображаться количество сгенерируемых элементов каждой группы, достаточно просто указать финальное число элементов в каждой группе producers.
  - Consumers будут иметь условия на подобии:
    - {A} | {B}
    - {A, B, C} и если нет 'A' и 'B' в очереди, то только {C}

Synchronization w/semaphores

Implement a FIFO queue which holds up to 16 elements. Each element is of a specified type (A/B/C) and has a value assigned (8-bit unsigned). The queue shall synchronize passing the elements between producer- and consumer- processes:

    There are three producer processes (one for each element type). Each producer creates between 64-128 elements of the corresponding type. No delays.
    There are two consumer processes:
    The first consumer consumes elements of type A or B (single elements, one by one). It is only allowed to consume these elements if the queue does NOT consist of element C.
    The second consumer consumes elements of all three types (at once) - if the queue consists of element C. If the queue consists of ONLY elements C then the consumer is allowed to consume elements C alone (one by one).
    The solution shall be implemented in C or C++, using processes, shared memory and semaphores in Linux (not necessarily Phoenix).
    Proper demonstration shall be prepared.
